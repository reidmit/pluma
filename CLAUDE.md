# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project

This repo (named `pencil` on disk) implements **Pluma**, a small statically-typed functional language. The CLI binary is named `pluma`, source files use the `.pa` extension. The language is documented under `site/content/docs/` (the zola-built docs site). Pluma has three backends sharing one IR: the **VM** (`pluma run`/`pluma test`/tests/bench ŌĆö the reference interpreter and differential oracle, not a deploy target), the **WasmGC** backend (the server/compute deploy artifact), and the **JS** backend (`pluma build --target browser` ŌĆö the browser/client deploy artifact). See "Workspace layout" and "Platform targets".

## Pluma syntax ŌĆö common gotchas

When writing `.pa` code, these are the traps that don't match other languages' intuitions. When in doubt, mirror a fixture in `tests/run/`.

- **Zero-arg calls use `()`, not `{}`.** `do-it ()`, `dict.empty ()`, `io.read ()`, `random.int ()`. `{}` is a block expression ŌĆö `do-it {}` parses as calling `do-it` with a block, which is a type error.
- **Pluma is uncurried.** `add 5` is an arity error, not partial application. To partially apply, wrap: `fun y { add 5 y }`.
- **Function literals: args bare before the brace.** `fun x { x + 1 }`, `fun x y { x + y }`. Zero-arg form is `fun { ... }`. Top-level functions: `def name = fun args { body }`.
- **Single-arg calls take no parens.** `print x`, `fact 5`, `to-string n`. Parens are only for grouping a sub-expression: `print (fact 5)`, `print (1 + 2)`.
- **Refutable `if`: `if subj is pat { ... } else { ... }`.** The `else` branch is the no-match case. Example: `if r is ok v { v } else { 0 }`.
- **`when` chains multiple `is` branches.** `when c is red { "r" } is green { "g" } is blue { "b" }`. No commas; only the first arm has the subject (`when c is ŌĆ”`), subsequent arms start with bare `is`.
- **Type annotations use `:: TYPE`.** Top-level: `def length :: fun (list a) -> int = built-in "list-length"`. Local: `let xs :: list int = []`. Parens wrap compound type args inside a `fun` signature (`fun (list a) -> int`, `fun (option int) -> int`).
- **Arithmetic operators `+ - * / %` are overloaded over int and float** via the `numeric` trait (and a heuristic for `%`); there are no dotted float operators. Both operands must be the same type ŌĆö `2 + 3.5` is a type error (no implicit int/float promotion). `<` `>` `<=` `>=` dispatch through `ord` in generic code, but on concrete `int`/`float` operands compile to direct comparison opcodes; for `float` that's IEEE-754, so a `NaN` operand makes all four relations `false` (matching structural `== !=`, also IEEE ŌĆö `nan != nan` is `true`). The total-order `ord.compare` used by `list.sort`/generic `ord` code is separate and still orders `NaN`. `== !=` are structural. `++` is string concat.
- **`??` unwraps an `option`/`result`** to a bare value, else its right-hand default: `opt ?? fallback`, `(dict.lookup m k) ?? 0`. Lazy (the default is only evaluated on `none`/`err`) and right-associative, so `a ?? b ?? c` chains. It's the recovering dual of `try` (which propagates the failure): `??` desugars to `option.or-else`/`result.or-else`, exactly as `try` desugars to `.then`. For `result`, the `err` value is discarded.
- **List literals support `...` spread.** `[1, ...xs, 2, ...ys]` ŌĆö a `...expr` element splices in another list (its element type must match). Spreads can appear at any position, any number of times; prefer this over `list.concat [head] tail`. Note the asymmetry with list *patterns*, where `...rest` is single and trailing-only (`[head, ...tail]`).
- **Record literals: field shorthand + update spread.** `{ name, age }` is shorthand for `{ name: name, age: age }` (the value is the in-scope binding of the same name). `{ ...base, name: "x" }` is record *update*: a copy of `base` with the listed fields overridden (`base` is unchanged ŌĆö records are immutable). Update is **update-only and type-preserving** (Elm-style): every override field must already exist on `base` at the same type, so the result has `base`'s exact type ŌĆö overriding a non-existent field or changing a field's type is a type error (use a plain `{ ... }` literal to build a different shape). The spread takes exactly one base and must come first (`{ ...base, fieldsŌĆ” }`); overrides may use shorthand (`{ ...a, name }`). Unlike list spread, a record literal allows at most one `...`.
- **String interpolation: `"$(expr)"`.** Non-string values need explicit `to-string`: `"n = $(to-string n)"`. A bare string variable interpolates directly: `"hi $(name)"`.
- **Duration literals: digits + a time unit, no space.** `1s`, `250ms`, `2m20s`, `3h2m10s`, `100d` are `duration` values (the prelude type behind `core.time`) ŌĆö no `use core.time` needed to write one. Units are `d h m s ms us ns`; segments must each appear at most once and in strictly descending order (`2m20s` ok; `20s2m`/`2m3m` are parse errors). Integer components only ŌĆö `1.5s` is rejected. The literal lowers straight to a constant, so it's not sugar for `time.seconds`; `time` is only needed to operate on the value. The formatter normalizes non-canonical spellings (`90s` ŌåÆ `1m30s`).
- **Enums.** Declare with newline-separated variants; payload follows the variant name:
  ```
  enum option a {
      some a
      none
  }
  ```
  Construct as `option.some 42` or bare `some 42` when the type is clear. Match by bare variant: `when x is some n { ... } is none { ... }`.
- **`def` is top-level; `let` is local.** Top-level bindings can't use `let`, and `let` patterns must be irrefutable (use `if`/`when` for `some`/`ok`/etc.).
- **`defer expr` runs cleanup at function exit.** A body statement (like `let`) that schedules `expr` to run when the enclosing **function** returns ŌĆö on the normal path *and* when a `try` short-circuits on failure. Multiple `defer`s run LIFO; a `defer` only fires if execution reached it (so one after a short-circuiting `try`, or in an un-taken `if` branch, is skipped). `defer` itself evaluates to `nothing`. It also fires across `await` suspension points and when a task is cancelled or fails (`tests/run/task-defer`, `scope-race`, `scope-deadline`). Two facets remain unbuilt: **awaited `task`-returning cleanup** (a `defer` whose expression is itself a task runs the thunk but doesn't *await* the resulting task) and block/`scope`-exit-scoped cleanup (`defer` is function-anchored by design) ŌĆö see `notes/ASYNC.md`.
- **Top-level defs are private by default; `public`/`opaque` widen that.** A bare `def`/`enum`/`alias` is visible only inside its own module ŌĆö importers can't see it (you'll get "`x` is private to module `y`"). Prefix `public` to export it (`public def map = ŌĆ”`, `public enum color { ŌĆ” }`). `opaque` is enum-only and exports the *type* while hiding its *constructors*: importers can name `mod.token` and pass values around, but can't `mod.token.mk` or pattern-match it ŌĆö make values through public smart-constructor functions instead. `opaque def`/`opaque alias`/`public trait` are parse errors (traits/instances aren't on the visibility ladder; instances are always exported). Visibility only gates the analyzer's cross-module type-checking ŌĆö codegen still compiles every def, so it never changes runtime behavior. The whole stdlib is explicitly `public`; mirror that when writing a module others import.
- **`task a` is the async carrier; `try` awaits it.** A `task a` is a cold, re-runnable async computation (build it with `task.return`/`task.fail`/`task.sleep`/`task.yield`, or by calling a function that awaits). `task` is the third `try` carrier: inside a function that returns a task, `try x = some-task` awaits it and the whole function is a `task` (an honest effect annotation ŌĆö there's no `async`/`await` keyword). **`core.task` is *not* auto-imported** ŌĆö unlike `option`/`result`, you `use core.task` to name any `task.*` function (`task.return`, the combinators, ŌĆ”). The async *syntax* still needs no import: `try`/`??` over a task, the `scope` block, `defer`, and duration literals are type-driven and lower to fully-qualified globals. But since you can't build a task without `task.return`/`task.fail`, async code imports `core.task` in practice. `?? `/`task.or-else` recover; `task.attempt` reifies failure to a `result`; `task.map`/`task.then` transform. The runtime is lazy ŌĆö a script that never makes a task pays nothing. **Structured concurrency is built:** `scope`/`manual scope` (keywords) with handle methods `s.spawn t` / `s.next ()` / `s.cancel ()` / `s.cancel-after d`, plus the combinators `task.both`/`all`/`settle-all`/`race`/`any`/`pool`/`with-timeout`/`retry` (all pure Pluma over the scope kernel). A scope handle can only be used where it's bound or in an `s`-capturing lambda, not passed as a plain function argument. `task.shielded` is built (`tests/run/task-shielded`), and `defer` cleanup fires across awaits and on cancel (see the `defer` gotcha above); the one async-`defer` facet still unbuilt is **awaited `task`-returning cleanup** ŌĆö see `notes/ASYNC.md`. Mirror a `tests/run/task-*` / `tests/run/scope-*` fixture when writing async code.
- **`use core.foo` for stdlib imports.** Available modules include `core.list`, `core.dict`, `core.bytes`, `core.string`, `core.math`, `core.assert`, `core.testing`, `core.hex`, `core.base64`, `core.random`, `core.uuid`, `core.time`, `core.task`. `ref`, `option`, and `result` are auto-imported ŌĆö don't `use` them. `core.task` is *not* (see the async bullet above): `use core.task` whenever you name a `task.*` function.
- **Tests are a library, not syntax.** There is no `test` keyword. A `*.test.pa` file exports `def tests :: testing.suite = fun t { ... }` and registers cases with `t.case "name" (fun { ... })` (also `t.group`/`t.skip`/`t.focus`/`t.todo`). A case body returns a `result`: `core.assert` checks (`assert.equals`, `assert.is-true`, ŌĆ”) each return `ok ()`/`err msg`, and `assert.all [..]` combines several. `pluma test` discovers the files and runs them. See `compiler/src/stdlib/*.test.pa` for examples.

For unfamiliar stdlib calls, `grep tests/run/*/main.pa` for a working example rather than guessing.

## Workspace layout

Cargo workspace (see `Cargo.toml`):

- `compiler/` ŌĆö the language frontend: tokenizer, parser, analyzer, types, diagnostics. The crate's public surface (`lib.rs`) re-exports `Compiler`, `Diagnostic`, `Module`/`ModuleExports`, `Tokenizer`, `Token`, and module-name/version constants; `ast` and `types` are `pub mod` (consumed by `ir` and codegen). Other modules (`analyzer`, `parser`, etc.) are private.
- `ir/` ŌĆö lowers the typed AST into a mid-level IR (`ir::lower(&compiler)` returns an `ir::IrProgram`). The IR is where backend-independent optimization (devirtualization, repr/box-unboxing, monomorphization, async CPS) happens, and it's the single source all three backends ŌĆö `codegen` (ŌåÆ VM bytecode), `wasm` (ŌåÆ WasmGC), and `js` (ŌåÆ JavaScript) ŌĆö consume.
- `codegen/` ŌĆö lowers the `ir::IrProgram` into VM bytecode (the **VM backend**). `codegen::compile_from_ir(&program)` returns a `vm::Program` ready to execute.
- `vm/` ŌĆö bytecode VM that executes the compiled program. `VM::new(program).run()` is the entry point. `print` writes through a configurable `StdoutSink` (process stdout by default; tests inject a `Buffer` sink). `vm::stdlib::register_compiler` seeds the analyzer with any Rust-defined native module types ŌĆö currently none, since every stdlib module (including `core.dict`) is a `.pa` source; the mechanism remains for any future module whose signature the `.pa` surface can't express. **Charter:** the VM is Pluma's *reference interpreter* and the *dev/test/bench engine* (`pluma run`/`pluma test`, the `tests/run` differential oracle, `bench`) ŌĆö it is **not** a deploy target. The VM corresponds to `Platform::Native`, which provides every capability so nothing is ever gated. Keep all three backends in lockstep: it's the cross-check (`conformance/`, which diffs WasmGC + JS against the VM oracle) that catches codegen bugs.
- `wasm/` ŌĆö the **WasmGC backend** (Pluma's deploy backend): lowers the same `ir::IrProgram` into a WasmGC module, `wasm::emit(&program) -> Result<Vec<u8>, _>`. Covers the full synchronous + async core plus `core.io` under `Platform::Server`. Capability gating per deploy target lives in `compiler::Platform` (see "Platform targets" below). Correctness is pinned by the `conformance` crate, which runs each emitted module in wasmtime and diffs stdout+status against the VM ŌĆö **full parity: 195/195 execution fixtures (`tests/run` + `tests/run-fail`) match, 0 diffs, 0 skips** (VM-internal negative tests like the unknown-builtin guard live in `tests/vm_errors.rs`, not the cross-backend corpus).
- `js/` ŌĆö the **JS backend** (Pluma's browser/client deploy backend): lowers the same `ir::IrProgram` into a self-contained JavaScript module, `js::emit(&program) -> Result<String, _>` (runtime preamble `js/src/runtime.js` + compiled functions + globals + entry). The IR is ANF and JS is uniformly boxed like the VM, so it's a near-1:1 translation (mirrors `codegen`'s traversal; `Repr`/`Box`/`Unbox` are no-ops). Synchronous core today (async `Await`/CPS, the `wire` codec, and full i64 via BigInt are follow-ons); ints are JS `number` (53-bit). Correctness is pinned by the `conformance` crate, which runs each emitted module under `node` and diffs stdout+status against the VM (159/195 execution fixtures, 0 diffs; `core.io` (gated) and async (unsupported) fixtures are skipped, and a small `denied` list tracks the rest ŌĆö see `CONFORMANCE.md`). `core.dom` (browser host: `dom.set-text`/`set-html`/`get-value`) is the first browser host surface; see `web/` for a DOM demo (`just web-demo`).
- `cli/` ŌĆö command dispatcher. `run`, `build`, `format`, `tokenize`, `analyze` are wired. `build` emits JS for the browser target (`--target browser`, `-o out`). `tokenize` and `analyze` are debug-build only (they dump Debug-format output of types whose Debug is gated on `debug_assertions`).
- `lsp/` ŌĆö language server, packaged for VS Code via the extension in `vsix/` and for Zed via the extension in `zed/`. Both editor extensions are thin clients that just launch `pluma-language-server`; all features (diagnostics, hover, formatting, highlighting via semantic tokens) live in the LSP so they're shared. The `zed/` crate ships no Tree-sitter grammar and is a standalone (non-workspace) cdylib built for `wasm32-wasip1`.
- `tests/` ŌĆö VM-only snapshot tests plus the IR tests. The Cargo package is named `tests`. **Four fixture suites, one job each:** `tests/run/` = happy-path VM execution (status `ok`; also the conformance corpus, and ŌĆö for fixtures carrying an empty `bench` marker file ŌĆö the compute-perf corpus); `tests/run-fail/` = programs that compile but fail at runtime (status `runtime error`; diffed across backends for *error parity*, excluded from perf); `tests/analyze/` = the frontend (parse/annotate/typecheck ŌĆö happy *and* unhappy, i.e. parse + type errors, rendered with source carets); `tests/format/` = formatter output + idempotence. Harness files live at the crate root (`tests/{analyze,run,format}.rs`); `run.rs` covers both `run/` and `run-fail/` via two `harness!` roots. Cross-backend correctness (WasmGC + JS vs the VM oracle) lives in the `conformance/` crate, not here. See "Testing" below.
- `conformance/` ŌĆö the **cross-backend conformance + perf harness** (the standing reference for all three runtimes). Runs each execution fixture (`tests/run` Ōł¬ `tests/run-fail`) through the VM (oracle), WasmGC (in wasmtime), and JS (under node), asserts the deploy backends match the VM, collects timing (a dev-loop compile-vs-exec breakdown over the non-`bench`-marked `tests/run` fixtures, plus a compute-throughput view over the `bench`-marked ones), and renders the report. `cargo test -p conformance` is the correctness gate; `cargo run -p conformance` (`just conformance`) regenerates the committed `CONFORMANCE.md` coverage doc; `--perf` adds perf tables ŌåÆ `target/conformance/report.md`. Backend runners: `src/{wasm_host,js_host,run}.rs`; support matrix + corpus in `src/lib.rs`.
- `bench/` ŌĆö microbench runner that times each `bench`-marked `tests/run/<name>/` fixture through the VM. Benchmarks live in `tests/run` (so they also get snapshot + cross-backend correctness coverage); an empty `bench` marker file in the fixture dir flags the compute-stress subset that `bench` and `conformance`'s compute-perf view select on.

## Testing

Fixtures live under `tests/<suite>/<name>/main.pa` (and optionally additional `.pa` files for multi-module cases). Each fixture has an `analyze.snap` / `run.snap` / `format.snap` next to its `main.pa` ŌĆö an insta snapshot file with a 3-line YAML header followed by the captured output. The suites divide by *what stage they pin*:

- **`tests/analyze/`** (frontend) ŌĆö runs the compiler frontend in-process and snapshots `{:#?}` of the typed `Module`, **or** formatted diagnostics (with a source excerpt + caret) on failure. This is where **unhappy** cases live: parse errors and type errors. Reject-at-compile-time fixtures belong here, *not* in `run/`.
- **`tests/run/`** (happy execution) ŌĆö compiles, lowers to bytecode via `ir::lower` + `codegen::compile_from_ir`, calls `vm::VM::run()` with a `vm::OutputSink::Buffer`, and snapshots a combined `status / stdout / stderr` block. **Status `ok` only** ŌĆö every fixture here is a successful run, so the suite doubles as the conformance corpus and the perf-exercise corpus.
- **`tests/run-fail/`** (runtime failure) ŌĆö same harness/snapshot shape as `run/`, but for programs that compile and then **fail at runtime** (status `runtime error`). Kept apart so `run/` stays clean for perf; still diffed across backends by `conformance` for error parity.
- **`tests/format/`** (formatter) ŌĆö formats the source, asserts idempotence (format twice = same), and snapshots the output. A curated, syntax-diverse corpus pinning formatter losslessness; a distinct concern from analysis.

The harnesses live in `tests/{analyze,run,format}.rs` (registered via `path =` in `tests/Cargo.toml` so they sit alongside the fixture directories rather than in a nested `tests/tests/`). `datatest-stable` generates one `#[test]` per fixture by scanning for `main.pa`; `run.rs` scans **both** `run/` and `run-fail/` (two `harness!` roots, one `run_fixture` fn). Tests set cwd to the workspace root so the `Module` Debug impl renders paths as `tests/analyze/<name>/main.pa` (portable across checkouts). VM-*internal* negative tests ŌĆö a malformed program that must surface a clean runtime error rather than panic, e.g. an unknown `built-in` tag ŌĆö live as plain `#[test]`s in `tests/vm_errors.rs`, **not** as fixtures: they aren't cross-backend cases (the deploy backends reject them earlier), so keeping them out of the corpus is what lets WasmGC hold full parity.

The `conformance` crate's cross-backend correctness corpus is `tests/run` Ōł¬ `tests/run-fail` (the deploy backends must reproduce the VM oracle on both happy and failing runs); its perf views both live in `tests/run`: the dev-loop benches the non-`bench`-marked fixtures (tiny programs), the compute view the `bench`-marked ones (the longer stress programs moved over from the former `benchmarks/programs`).

When changing analyzer/parser output or VM behavior, regenerate snapshots with `cargo insta review` (interactive accept/reject) or `just test-write` (accept all). Don't hand-edit `.snap` files. To add a new test, create the fixture directory + `main.pa`, run `just test-write`, and review the generated snapshot.

## Compiler architecture

`Compiler` (in `compiler/src/compiler.rs`) is the orchestration entry point. It owns a `HashMap<String, Module>`, a `HashMap<String, ModuleExports>` cache (see below), and a `Vec<Diagnostic>`. It exposes `tokenize()` and `check()`. `check()` runs through parsing + type analysis and returns the typed entry `Module`; the bytecode/run pipeline lives downstream in the `codegen` and `vm` crates.

`check()` does a DFS load (`load_module`): for each module, it parses, then recursively loads anything in its `uses`, then analyzes the module itself (so dependencies are always analyzed first). A `visiting` set catches import cycles and reports them as a `Diagnostic`. After analysis, the module's `exports: Option<ModuleExports>` is populated from its top-level defs and cached for any later importer.

`ModuleExports` (in `module.rs`) is a three-way map: `values` for top-level value defs (and alias constructor functions), `aliases` for alias type defs (name ŌåÆ resolved type), `enums` for enum type defs (name ŌåÆ ordered list of variants). The analyzer's `set_imports` takes both this and a parallel local-name ŌåÆ fully-qualified-module-name map so qualified enum types can be reconstructed at use sites.

Visibility (`ast::Visibility`, set by the parser from a leading `public`/`opaque`) is applied when the analyzer *builds* exports (`analyzer.rs`): a private def is omitted from `values`/`aliases`/`enums` and its name recorded in a `private` set (so importers can report "`x` is private" rather than a bare "not found"); an `opaque` enum is exported in `enums` with `param_count` intact but an **empty variant list**, so the type resolves while construction/pattern-matching find no variants. So filtering happens once, at the export boundary ŌĆö consumers just see (or don't see) the names.

Pipeline within a single module:

1. **Tokenize** ŌĆö `Tokenizer::from_source(&bytes)` (in `tokenizer.rs`) yields `Token`s from raw source bytes.
2. **Parse** ŌĆö `Parser::new(bytes, tokenizer).parse_module()` produces a `ModuleNode` (which has both `uses: Vec<UseNode>` and `body: Vec<DefinitionNode>`), plus a comment map and `ParseError`s. The AST is split across `ast/*.rs` with one file per node category (`call`, `definition`, `enum`, `expr`, `fun`, `if`, `let`, `literal`, `pattern`, `use`, `when`, `while`, etc.) and a shared `ast/mod.rs` re-export list.
3. **Analyze** ŌĆö `Analyzer::analyze(module)` in `compiler/src/analyzer.rs` is a Hindley-Milner-style three-phase pass:
   - `constrain(ast)` walks the AST, assigning fresh type variables and producing `Constraint`s; literals get concrete types inline.
   - `unify(&constraints)` solves for a `Substitution`.
   - The substitution is applied back to the AST, filling in the inferred types in-place (the AST is mutated).

   Built-in types (`int`, `bool`, `string`, `regex`, `float`) are seeded into the type scope at the start of `analyze`. Value/type scopes are stacks of `HashMap`s; identifiers resolve into `ValueBinding` / `TypeBinding` (in `binding.rs`). The analyzer also takes an `imports` map (local namespace name ŌåÆ that module's exports) via `set_imports`.

Errors flow through a single `Vec<Diagnostic>` threaded by reference through tokenize/parse/analyze. `Diagnostic` carries optional module + range and is rendered by `cli/src/printing.rs`. The error kinds live in `compiler/src/errors/` (`parse_error.rs`, `analysis_error.rs`, `usage_error.rs`).

## Platform targets

There is **one** WasmGC backend; "server" and "browser" are **platforms** = host-capability *profiles*, not separate backends. A `Platform` (`compiler/src/platform.rs`, re-exported as `compiler::{Platform, Capability, module_capabilities}`) decides which stdlib modules a program may `use` and, downstream, which host imports the `.wasm` declares.

- `Capability`: `Fs, Stdio, Net, Clock, Entropy, Process, Dom, Fetch, Timer`.
- `Platform`: `Native` (default ŌĆö the VM; provides **all** caps so nothing is ever gated), `Server` (`Fs/Stdio/Net/Clock/Entropy/Process`), `Browser` (`Dom/Fetch/Timer/Stdio`).
- `module_capabilities(name)` ŌĆö only gated modules need a row (today just `core.io ŌåÆ [Fs, Stdio, Process]`); **unlisted modules and every user module default to `&[]`, never gated.**

Gating is a **compiler-frontend** concern only: a `use` of a module whose caps the active platform lacks is a compile error at the `use`-statement span, emitted in `Compiler::load_module`'s import-rejection loop. `ir::lower` and `wasm::emit` need no platform ŌĆö by the time they run, the analyzer has already rejected disallowed `use`s and the wasm import set is fully determined by the reachable builtins. Select a platform with the `Compiler::with_platform(self, p)` builder (the ~14 existing constructor call sites stay on `Native`, so default builds and all snapshots are unchanged). This whole-module gating is the foundation the future `server def` islands (`notes/FULLSTACK.md`) build on.

## Type system notes

- **Enums are nominal, with module-qualified identity.** `Type::Enum(String)` carries a fully-qualified name `<defining-module>.<enum-name>` (e.g. `colors.color`). Structural variant info lives in `Analyzer::enum_defs`, keyed by that same qualified name. Both locally-defined enums and imported ones live in `enum_defs` under their qualified keys, so variant resolution is uniform. `Type`'s `Display` strips up to the last dot, so snapshots and error messages still render bare names (`color`).
- **`FieldAccess` is overloaded** in `constrain_expr`. The handler checks, in order: chained `module.enum.variant` (cross-module variant access), imported-module namespace lookup (`module.value`), local enum variant access (`enum.variant` via the type_scope binding's qualified name), then falls back to a `PartialRecord` constraint for actual record fields.
- **`module.TypeName` in type positions.** `TypeIdentifierNode` carries an optional `module: Option<IdentifierNode>` prefix populated by `parse_type_identifier`. `Analyzer::type_expr_to_type` resolves it against the importer's `imports` map (checking exported enums first, then aliases) and produces the qualified `Type::Enum` or the resolved alias type.
- **Variant patterns disambiguate by subject type.** `resolve_variant_pattern` looks up a name in the subject's enum when the subject type is concrete; otherwise it does a global lookup across `enum_defs` and reports `AmbiguousVariant` if more than one enum has that name. The global pool includes imported enums too, so bare variant patterns can match imported enums when the subject type is known.
- **`when` is exhaustiveness-checked** in `annotate_expr` (post-substitution, so the subject type is known). `Type::Bool` and `Type::Enum` are checked structurally; other subject types currently rely on a `_` catch-all and are otherwise skipped.
- **`let` bindings with concrete-typed values bind monomorphically** (`Scheme::Forall(vec![], ty)`) rather than going through Gen/Inst. Lets the value's resolved type (e.g. `Type::Enum(color)`) be visible at constraint-gen time for downstream pattern resolution. Polymorphic values still take the Gen/Inst path.
- **Cross-module values are instantiated per use** via `Analyzer::instantiate`, which walks the imported type and replaces every free `Type::Var` with a fresh one. Gives per-call-site polymorphism for imported functions.
- **The unifier has an `Enum ~ Enum` case** that succeeds iff the qualified names match. `Module`'s `Debug` impl prints comments through a `BTreeMap` so the test snapshot output is stable.

## Conventions

- **Tabs**, not spaces. Run `cargo fmt` (project uses `rustfmt.toml`).
- Diagnostics are accumulated, not raised ŌĆö pass `&mut Vec<Diagnostic>` rather than returning `Result` for non-fatal issues.
- AST node types are split one-per-file under `ast/`; add new nodes by adding a module and re-exporting from `ast/mod.rs`. The `r#use` and `r#enum` files use the raw-identifier prefix because `use` and `enum` are Rust keywords.
- `Module::ast` is an `Option<ModuleNode>` populated by `parse`. The analyzer mutates the AST in place to attach inferred types ŌĆö don't clone it out.
- `Compiler` derives `Debug` only in debug builds (`#[cfg_attr(debug_assertions, derive(Debug))]`); `analyze` output uses `{:#?}` of the `Module`, so `Debug` impls for AST nodes are part of the test surface.
